Classypants

Overview

Classypants is a tool to help make sense of the values of $PATH like environment variables. The following screenshot should give an idea of how a classypants window looks like.

As you can see, the window is pretty simple. There is one menu (\m/) at the top left, a search bar at the top and the list of entries as the main focus. A few entries are highlighted in red, which indicates that they don't exist on the file system.

Download

Download the latest jar here. Or browse the files on the bitbucket project page.

Basic Usage

Once you open classypants, you need to get the value of your $PATH variable into classypants. Currently, the only way to do this is via the clipboard. Copy your $PATH variable's value and go to the menu at the top left and click on Paste or just hit C-S-v. Classypants will sanitize and show you the content in a human-consumable way.

You can search for entries you want to find by entering the search string in the top input box and hit the Return key. The search box is quite powerful and although not immediately apparent, features a lisp-backed syntax to do complex queries on your entries.

Note that a Copy menu item is also present, which, well copies a $PATH like value with the current entries. This might be more useful, once editing features are added to Classypants.

Advanced Search

Classypants' advanced search is the thing I love about it. It might seem hard and even ridiculous, but if you let it sink it, it will feel quite natural and you might even want it every where :). That said, if you have a slight knowledge of clojure the language, you'll be right at home with this syntax.

Simple searching

You can just type a word in the search input, hit Return and expect all entries that contain this word to be matched. For example,

resources

Please note that whatever you enter in the search bar, needs to be a valid clojure atom. Be it a symbol/string/number/list or anything else. If you don't know what it means, don't worry. It just means that in case you are entering anything non-alphanumeric in the search bar, you'll almost certainly need to enclose it in " characters. For example, filtering for all that have .jar in their value, you have to search for ".jar", not .jar. If you are doubtful as to whether you need the quotes, it is safer to put them in. No harm done.

Negate match

Your first search modifier. If you want to match all entries that don't match your search string, you'll use the :not modifier.

:not ".jar"

will match all entries that do not contain .jar

Combine searches

Similar to the :not above, you also have :and and :or to mix and match against your entries.

lib :and ".jar"

will match entries that have both lib and .jar in them.

Can Haz?

Experimental

Using the :has directive, you can match the entries that point to directories (or archives) that contain the said file or directory. For example, the search string,

:has my-awesome-resource.xml

will filter out the directories that contain this file my-awesome-resource.xml present inside them.

Inspiration

I once had to (and still have to, occasionally) deal with extremely long CLASSPATH variable values, which were built with large shell scripts and it included many non-existent jars/directories, many ones that were of the same third party library, but of different versions, you get the point. I needed a way to get a bird's eye view of the CLASSPATH. Initially, I used to just do

$ echo "$CLASSPATH" | tr : '\n' | grep 'what-am-i-looking-for'

But, quickly enough, I wanted more. I needed to find the first entry in the CLASSPATH that contained a particular class file. Now, I could go ahead and do that in the shell too, but I decided I'd make a tool to give a visual appeal to these rather lifeless variables. Moreover, I needed a reason to make something in clojure :)
